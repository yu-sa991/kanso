# ★アプリタイトル★
『「まだ大丈夫」を許さないアプリ』

# 1. サービス概要
『「まだ大丈夫」を許さないアプリ』食事記録の「めんどくさい」をすべて断捨離。1秒・3択のざっくり選択で、忙しい毎日にダイエットの意識づけを定着させ、運動など本来の健康管理に時間を使えるようにする振り返りアプリ。

# 2. このアイデアはどこから生まれたか

 2-1. 自分がダイエットアプリを使って日々の食事やカロリーを入力していた際、あまりにも入力項目が多すぎて嫌になり、途中で挫折して辞めてしまった経験から生まれました。

 詳細：健康になりたいと思って始めたはずなのに、毎食のメニュー検索やグラム数の入力に時間を取られ、「これではダイエットをしているのか、文字入力をしているのかわからない」と本末転倒なストレスを感じました。「もっと大まかでいいから、1秒でサクッと振り返りができるアプリがあれば続けられたのに」という、当時の自分が一番欲しかった切実な願いがこのアイデアの原点です。



【原体験のまとめ】
「痩せて健康になりたい」と思い、ダイエットアプリを使って日々の食事記録を始めました。最初はモチベーションも高く、毎食のメニューや分量を一生懸命に入力していました。しかし、入力項目があまりにも多すぎたため、次第に「健康管理のための時間」よりも「スマホに向かって入力する時間」のほうが圧倒的に長くなってしまったのです。その結果、負担に耐えきれなくなり、アプリを開く頻度が減り、最終的には記録すること自体をやめてしまいました。

【生まれた感情】最初は前向きだった気持ちが、いつの間にか「今日も入力しなければならない」という義務感へと変わり、日々の大きなストレスになっていきました。アプリから足が遠のくにつれて、「また継続できなかった」「自分は意志が弱いのではないか」という強い罪悪感や敗北感を抱くようになりました。データを入れること自体に疲れ果て、本来楽しむべきダイエットへのモチベーションまで完全に消えてしまったのです。

【覚えた違和感】
ここで強い違和感を覚えました。私は「ダイエット（健康になりたい）」のであって、「完璧な食事データベースを作りたい」わけではありません。自分を助けてくれるはずの便利なツールが、むしろ自分の時間とエネルギーを奪う足枷になっているという「本末転倒な状況」に矛盾を感じました。毎日3食、何十項目も正確に入力し続けられる人ばかりではないはずです。今のアプリの設計は、人間のリアルな継続力を無視した、ユーザーへの配慮が足りないシステムなのではないか、という疑問が残りました。

[自分自身の体験と違和感]
健康になりたいと思って始めた既存のダイエットアプリでしたが、毎食のメニュー検索やグラム単位のカロリー入力を繰り返すうちに、「スマホに向かって入力する時間」のほうが長くなってしまい、**いつの間にか入力すること自体が目的になってしまっている本末転倒な状況**に強い違和感を覚えました。
自分を助けてくれるはずの便利なツールが、むしろ自分の時間とエネルギーを奪う足枷になっている矛盾を感じ、「がんばるユーザーほど入力疲れで燃え尽きてしまうシステム」から脱却したいと考えたことが開発のきっかけです。

2-2. なぜそれが「気になった」のか

自分の性格：極度のめんどくさがり。
過去の経験：細かな食事入力に時間を取られすぎてしまい、挫折した。
大切にしている価値観：入力ではなく、運動など「本来のダイエット」に時間を使いたい。だから記録は大まかでいいのではないかと思いました。

# 3. 課題の整理（表面的になっていないか）
この入力方法で本当にダイエット達成できるのか。

3-1. 表に見えている困りごと

その体験の中で、最初に感じた「困りごと」は何ですか？

【一言で言うと】「カメラ撮影で手軽になったはずなのに、結局は手入力の手間から解放されないストレス」

【困りごとの詳細】

撮影できない場面の存在：
外食先でのビジネスランチ、盛り付けが汚れてしまったとき、撮影を忘れて食べ終えてしまったときなど、日常には「カメラを向けられない・撮れないシーン」が多々ある。

手入力へ逆戻りする負担：
撮影できなかった瞬間、結局は過去のアプリと同様に「メニュー名や分量を細かく思い出しながら打ち込む作業」が発生してしまう。

手軽さの裏にあるギャップ：
「写真を撮るだけ」という手軽さを期待して始めたからこそ、たまに発生する手入力の面倒くささがより際立ち、大きなストレス（挫折の引き金）になる。

この課題は「機能（カメラ）の限界が、ユーザーのやる気を削いでいる」着眼点

3-2. 本当に解決したい課題は何か

「なぜそれが困るのか？」

- 時間がとられるから
- 短時間で入力できないから
- 入力が多すぎるとモチベーションが下がるから
- モチベーションが下がると困るのか？
　　　　　→食事を記録すること自体に疲れ果てて、本来の目的である「ダイエット（健康管理）」を諦めてしまうから。

→ 本当に解決したい課題：「面倒な記録作業のせいで心が折れ、本来やりたかったダイエットそのものを諦めてしまう、人間の『燃え尽き（挫折）』をなくしたい」


# 4. 想定ユーザーについて

4-1. 想定しているユーザー
仕事や、家事、育児などで時間を取られて、あまり時間がとれないけれど、ダイエットがしたい人向け

年齢設定：
  - 年齢：38歳・女性（既婚・共働き・子供2人）
  - 理由：アラフォーになり、基礎代謝が落ちて明らかに身体の線が崩れてきた。若い頃のダイエット法が通用しなくなり本気で焦りを感じる年齢だから

生活・状況設定：
  - 生活・状況：
  平日は朝から弁当・朝食を作り子供の送り出し、フルタイム勤務の会社ワーク、18時半にヘトヘトで帰宅した後は怒涛の夕食作りと家事に追われ、自分のケアは常に後回し。

  - 理由：
  日中のストレスと疲れから、夜静かになった部屋で「今日のご褒美」と無意識に高カロリーなお菓子をつまんでしまう。体重計に乗る勇気がなく、「まだギリギリ大丈夫」「明日少し動けばいい」と、現実から目を背けて都合よく錯覚してしまう自分に歯止めをかけるため。自分の「本当の食事量」を直視し、勘違いを正す必要がある。

環境設定：
  - 環境：
  仕事、家事、育児のタスクが詰まりすぎており、自分のために使える時間は1日5分すらない。スマホをじっくり触れるのは「通勤電車の中」か「布団に入って目を閉じる直前」だけという極限の環境。

  - 理由：
  1秒でも惜しい生活の中で、写真撮影や細かな文字入力は100%挫折する。「1秒・1タップ」のわずかなアクションだけで、自分の今のヤバさ（現状）を客観的に把握し、勝手な錯覚を打ち砕く必要があるため。
  

4-2. 自分とユーザーの距離
- 自分とユーザーの距離：
「ターゲットはまさに、過去の自分であり、今現在の自分そのものです」

理由・詳細：
これまで何度も「今度こそは」とダイエットアプリを使い始めては、あまりの入力項目の多さに力尽き、挫折を繰り返してきました。そして今現在も、「ダイエットのために食事入力を続けたい、現状を把握したい」という強い気持ちはあるのに、日々の忙しさに追われて結局入力が続かないという悪循環から抜け出せていません。だからこそ、このアプリは「綺麗事の理想」で作るのではなく、「めんどくさがりで時間がない自分自身が、今度こそ本当に使い続けられる最後の砦」として開発しています。

4-3. 実際に使われる可能性について(需要・存在確認)

- まず最初に使ってくれそうな人は誰か？　　
「本気でダイエットしたいけれど、細かい食事記録を続けるのは絶対に無理（または過去に挫折した）」という、ずぼら・タイム貧乏な人。

- その人は、どんな場面でこのアプリを思い出すか？　有名な食事管理アプリをダウンロードしたものの、初日から「メニュー検索やグラム数の入力」の多さにうんざりし、「もっと大まかでいいから、秒で終わるアプリはないの？」と嫌気がさした瞬間。
　
- 今の生活の中で、代わりに使っているものは何か？　入力項目が多すぎる既存の有名ダイエットアプリ（または、挫折して何も使わず現実逃避している状態）。

- それを置き換えてまで使う理由はあるか？　　　
　「1秒・3択」という圧倒的な手軽さでありながら、体重やカロリー、1ヶ月のカレンダーといった「現状把握に必要な最低限のデータ」が自動でしっかり手に入るから。


 4-4. ユーザーがサービスを導入して利用するまでのイメージ(導入・継続の流れ)
　　
① 認知・きっかけ（どんな場面で知る・思い出すか）
既存の食事記録アプリの入力の多さにめんどくさくなり、「もっと簡単に、大まかに記録が取れて現状を把握できるアプリはないか」と検索したり、SNSで同じ悩みを持つママの口コミを見たことがきっかけで出会います。

② 最初の一歩（登録・導入でユーザーがすること）アプリを起動後、最初の1回だけ【体重・名前・身長】を入力します。これだけで、アプリ側が自動でユーザーに最適な「標準体重」「1日の目標摂取カロリー」を計算し、メイン画面にわかりやすく表示してくれます。

③ 利用を続ける理由・仕組み（習慣・必要性・価値）一度使って終わりにならず、毎日使い続けたくなる理由は以下の3つです。

1.徹底的な「手間の断捨離」による継続（習慣）
日々の入力は「少ない・普通・食べ過ぎた（ヤバい）」の3択を選ぶだけ。体重は「＋ー」の簡易入力。布団に入って目を閉じる直前でもできる手軽さだからこそ、行動が「習慣」になります。

2.現実を突きつける「データの必要性」（価値）
1タップするだけで、1ヶ月の色のカレンダーや記録表が自動で生成されます。自分の「勝手な大丈夫という錯覚」を色、数字とビジュアル（太るキャラクター）で裏付けしてくれるため、「現状を直視するために毎日開かなければ」という必要性が生まれます。

3.モチベーションを高める「癒やしのUI」（楽しさ）
画面を開いたときに、きれいでかわいく、心が癒やされるデザイン（UI）に触れられるため、忙しいフルタイムママにとって、1日の終わりに少しだけホッとできる「お気に入りのお留守番キャラクター」のような存在になり、愛着を持って使い続けられます。

# 7. このアプリで実現すること
ダイエットアプリの入力を無理なく続けられるようにし、大まかな記録と現状把握を可能にすることで、ユーザーのダイエット継続を支える。


MVP（初期リリース）で実装する機能

「まずは1タップで記録ができて、大まかなデータが溜まる」というコア機能だけに絞ります。

1.初期設定機能：
初回のみ【名前・性別・年齢・身長・初期体重】を入力。
  ※性別は1タップで選択可能。選んだ性別（男女）に合わせて、裏側で正確な自動計算式を切り替えます。
　 ※自動計算ロジックを正確に働かせるため、性別は「男性/女性」の2択とし、年齢の入力も必須とします。


2.目標自動計算機能：
入力された情報から「適正な標準体重」と「1日の目標摂取カロリー」を自動算出・表示。
入力されたプロフィールを基に、裏側のプログラム（ハリス・ベネディクト方程式）が「標準体重（BMI 22）」と「1日の目標摂取カロリー（消費カロリー - 300kcal）」を自動算出。
    ※体重が更新されると、目標カロリーも自動で最適な数値に厳しくアップデートされ、都合の良い錯覚を防ぎます。

一日の目標カロリー計算式
アプリに組み込む「1日の目標カロリー」の計算式医学的・「ハリス・ベネディクト方程式（日本人版）」という計算式、裏側のRailsプログラムに以下の「足し算と掛け算」を1行書くだけで自動計算してくれます。
① まず「基礎代謝（生きてるだけで消費するカロリー）」を出す
🙋‍♀️ 女性の計算式：9.247 × 体重(kg) ＋ 3.098 × 身長(cm) － 4.33 × 年齢 ＋ 447.593
🙋‍♂️ 男性の計算式：13.397 × 体重(kg) ＋ 4.799 × 身長(cm) － 5.677 × 年齢 ＋ 88.362
② 「1日の活動量」を掛けて、キープできるカロリーを出すヘトヘトなフルタイムママ・パパは、デスクワークや家事育児で「標準的な活動量」があるため、上記の基礎代謝に 1.5 を掛けます。1日の消費カロリー ＝ 基礎代謝 × 1.5
③ そこから「目標カロリー（ダイエット用）」を設定
ここがアプリの真骨頂です。キープできるカロリーをそのまま目標にしたら痩せません。
健康的に痩せるための王道である「1日マイナス200〜300kcal（おにぎり1個分を我慢する）」を自動で引き算した数値を、最終的な「1日の目標カロリー」として画面に突きつけます。
最終的な目標カロリー ＝ 1日の消費カロリー － 300kcal

標準体重は、日本肥満学会などの基準に基づいた【BMI 22（最も病気になりにくく健康的な体重）】を表示する形になります。
あえて「美容体重」ではなく、医学的に最も信頼性の高い「標準（健康）体重」を身長から
自動計算（身長m × 身長m × 22）

# 8. このアプリの懸念点とその対策
 懸念点①：市場に同じような競合アプリ（類似アプリ）が登場すること

- 何が問題になりそうか：
日々、数多くのダイエットアプリがリリースされる中で、自分のアイデアと似たような「手軽さ」や「3択入力」を謳う競合アプリが登場し、埋もれてしまう可能性がある。

- なぜそう思うか：競合調査（カナヘイ、SmartDietなど）の結果、世の中には「可愛いもの」や「シンプルなもの」がすでに多数作られており、日々進化していることを実感したから。

- そのために考えている対策：
他社が「手軽さ」を真似してきたとしても、私のアプリは「とにかく入力を少なくする工夫」を極限まで徹底して差別化します。例えば、ただ3択を選べる機能（食事の多い・普通・少ない）だけでなく、体重の入力すら数字を打ち込ませずに「＋/ー」だけで完結させたり、文字の保存ボタンすら無くして「タップした瞬間に画面が閉じて記録完了する」といった、圧倒的な“ずぼらさ（1秒完結）”のギミックと使いやすさで、他のどのアプリよりも「これなら続けられる！」という独自のポジションを死守します。

懸念点②
何が問題になりそうか：入力が「1タップの3択だけ」と極限までシンプルなため、操作が単調になり、数週間でユーザーが「飽きてしまう」可能性がある。

なぜそう思うか：めんどくさがりなユーザーは、少しでも「毎日同じ作業の繰り返し」だと感じると、アプリを開くこと自体の優先度が下がってフェードアウトしがちだから。

そのために考えている対策：アプリ内に「クスッと笑える育成ゲーム要素」を取り入れます。その日の判定（🟢🟡🔴）に応じてキャラクターが放つ「お小言・褒め言葉（セリフ）」のバリエーションを豊富に用意します。毎日の1タップの瞬間に「今日はどんなセリフを言うかな？」という小さなワクワク感（アメとムチ）を用意することで、単調な記録作業を「楽しいルーティン」に変え、飽きを徹底的に防止します。

懸念点③：
日々の入力が「少ない・普通・食べ過ぎた」の主観による選択なため、本当は食べ過ぎなのに自分に甘く「普通」を選んでしまい、ダイエットの成果に繋がらない事例が発生すること。

なぜそう思うのか：
人間は、自分にとって都合の悪い現実に対して「勝手にごまかし（現実逃避）」をしてしまう性質があるから。

そのための対策：
本リリースで【感情を動かす！キャラクターのお小言＆褒め言葉機能（「普通」おねだりアラート搭載）】を実装します。ユーザーが「普通（🟡）」を4日連続で選んだ場合、システムが自動検知してキャラクターがジト目で「本当にきっちり普通だった？」と疑う特別ポップアップを表示。ユーザーに入力の手間を一切増やさずに、主観の甘さをユーモラスに軌道修正します。

2. ログイン機能（Reactでの情報保持・注意点・セキュリティー）について

① 画面リロード（再読み込み）時の記憶喪失対策
画面をリロードした際にログイン状態がクリアされてしまうのを防ぐため、ログイン成功時に発行される認証トークンをブラウザの「localStorage」に保存し、リロード後もログイン状態が維持される仕組みを実装します。

② URL直接入力による不法侵入対策（ルーティングの保護）
未ログインユーザーがURLを直接打ち込んでメイン画面にアクセスできてしまうのを防ぐため、React Routerを用いて「未ログインなら強制的にログイン画面へリダイレクトさせる」という保護されたルート（RequireAuthによる自動ドアの仕組み）を実装します。

③ ログアウト時のデータの消し忘れ対策（クリーンアップ）
ログアウト処理時には、React側のログイン状態（State）や localStorage に保存したトークンを完全にクリア（clear/初期化）し、前ユーザーの情報やキャッシュが残るバグを防止します。

【セキュリティーに関する今後のロードマップ】
セキュリティーの重要性を考慮し、本リリースに向けては、XSS（不正プログラム攻撃）によるトークン奪取を防ぐ、より強固な「HttpOnly Cookie」での認証への移行を目指します。
ただし、初学者のため実装の難易度による挫折を防ぐべく、まずはカリキュラム通りに `localStorage` を使って全体のモックアップを完成させ、その後、講師の方々やAIのサポートを受けながら段階的に HttpOnly Cookie へ書き換えていく方針で進めたいと思います。

10-3.キャッチアップ不足の懸念
 懸念点①：ReactとRailsの「データ連携」の難易度が想像できず、タスクが大幅に遅れる

 具体的な内容：
 React（フロントエンド）とRails（バックエンド）を別々に作る「API開発」の難易度を甘く見てしまい、「画面のボタンを押したらデータが保存される」という1日で終わると思った連携タスクに、エラーが多発して1週間以上かかってしまう懸念。

 それに対する対策：
 いきなり合体させようとせず、まずはReactだけでデータが動く「モックアップ（見た目の試作品）」を作り、次にRailsだけで動く「API」を作る。「画面」と「裏側」を完全に切り離して小さなステップに分解して開発することで、難易度をコントロールし、遅れを最小限に防ぎます。


 懸念点②：データベース（MySQL/Aiven）の無料枠の仕組みを誤解し、後からデータが消えたり作り直しになる

 具体的な内容：
 RenderやAivenの無料枠のルールや接続設定（環境変数など）のキャッチアップが足りないまま進めると、間違った設定のせいで「本番環境にデプロイした途端に動かない」「ユーザーのデータが保存されない」という事態になり、システム構成を最初から作り直すことになる懸念。

 それに対する対策：
 開発の最終段階で初めてデプロイ（公開）するのではなく、「3択ボタンが1つだけある画面」ができた段階など、極限までシンプルな状態で一度RenderとAivenへのデプロイを試す（早期デプロイ）。早い段階でインフラの仕組みをキャッチアップし、後からの大がかりな作り直しを未然に防ぎます。

細かな食事メニューの検索や、グラム単位のカロリー入力を毎日繰り返す中で感じた、「健康になりたいのであって、完璧な食事データベースを作りたいわけではない」「いつの間にか入力すること自体が目的になってしまっている」という本末転倒な違和感（原体験）から生まれた、忙しいアラフォー世代のための超ずぼら向けダイエット振り返りアプリです。がんばるユーザーほど入力の手間によって燃え尽きてしまう既存アプリの課題を、手間の徹底的な断捨離によって解決します。ターゲットは仕事、家事、育児に追われ、自分のケアが常に後回しになっているフルタイム勤務の38歳ママ・パパ。体重計に乗る勇気がなく「まだ大丈夫」と現実から目を背けて都合よく錯覚している極限の環境に寄り添います。主な機能は、アプリを開いて1秒で完結する「少なすぎ（🟢）」「普通（🟡）」「食べすぎ（🔴）」の巨大3択ボタンによる食事レコーディングです。ボタンのすぐ下には「腹八分目」「夜抜き」「間食お菓子」といったリアルな日常の客観的一言目安を常時表示し、主観による現実の「ごまかし」をシステム側からシャットアウトします。本リリースでは、ユーザーが都合よく「普通」を4日連続で選ぶとシステムが自動検知し、キャラクターがジト目で「本当にきっちり普通だった？」とユーモラスに疑う特別なポップアップアラート（連続「普通」おねだりアラート）を搭載。ユーザーに新しい測定や文字入力の手間を1ミリも増やすことなく、クスッと笑いながら「現実を直視」させ、本来の健康管理行動への意識づけを定着させます。

# 12. 追記
 一日の目標カロリー計算式
 アプリに組み込む「1日の目標カロリー」の計算式医学的・「ハリス・ベネディクト方程式（日本人版）」という計算式難しそうに見えますが、裏側のRailsプログラムに以下の「足し算と掛け算」を1行書くだけで自動計算してくれます。
① まず「基礎代謝（生きてるだけで消費するカロリー）」を出す
🙋‍♀️ 女性の計算式：9.247 × 体重(kg) ＋ 3.098 × 身長(cm) － 4.33 × 年齢 ＋ 447.593
🙋‍♂️ 男性の計算式：13.397 × 体重(kg) ＋ 4.799 × 身長(cm) － 5.677 × 年齢 ＋ 88.362
② 「1日の活動量」を掛けて、キープできるカロリーを出すヘトヘトなフルタイムママ・パパは、デスクワークや家事育児で「標準的な活動量」があるため、上記の基礎代謝に 1.5 を掛けます。1日の消費カロリー ＝ 基礎代謝 × 1.5
③ そこから「目標カロリー（ダイエット用）」を設定
ここがアプリの真骨頂です。キープできるカロリーをそのまま目標にしたら痩せません。
健康的に痩せるための王道である「1日マイナス200〜300kcal（おにぎり1個分を我慢する）」を自動で引き算した数値を、最終的な「1日の目標カロリー」として画面に突きつけます。
最終的な目標カロリー ＝ 1日の消費カロリー － 300kcal


標準体重は、日本肥満学会などの基準に基づいた【BMI 22（最も病気になりにくく健康的な体重）】を表示する形になります。
あえて「美容体重」ではなく、医学的に最も信頼性の高い「標準（健康）体重」を身長から自動計算（身長m × 身長m × 22）して突きつけることで、ユーザーの「まだ大丈夫」という都合の良い錯覚を客観的な数値で打ち砕く演出といたします。

### 🗃️ 各テーブルのカラム詳細

#### 📘 users テーブル（会員名簿ノート）
アプリのログイン認証、および目標カロリー・標準体重をユーザー個人の身体データから正確に自動算出するための基本プロフィールを管理します。
- **id** : `integer` / 主キー（ユーザーを一意に識別するための背番号）
- **name** : `string` / ユーザーの表示名
- **email** : `string` / ログイン用メールアドレス（重複を防ぐユニーク制約）
- **password_digest** : `string` / 暗号化されたパスワード（セキュリティーを考慮したハッシュ化データ）
- **gender** : `string` / 性別（「男性 / 女性」の2択。男女で異なるカロリー計算式を切り替えるため必須化）
- **age** : `integer` / 年齢（ハリス・ベネディクト方程式による目標カロリー算出用に必須化）
- **height** : `float` / 身長（cm / 標準体重・基礎代謝の自動計算用）
- **initial_weight** : `float` / アプリ開始時の初期体重（kg / 基礎代謝の計算用）

#### 📙 meal_records テーブル（食事チェックノート）
ユーザーが日々ポチポチした3択の判定データを、カレンダー形式にマッピングして蓄積していくためのテーブルです。
- **id** : `integer` / 主キー（食事記録1件ごとの通し番号）
- **user_id** : `integer` / 外部キー（どのユーザーの記録かを特定。usersテーブルのidとガッチリ紐付け）
- **date** : `date` / 食事の記録日（色分けカレンダーへのマッピング・判定用）
- **status** : `integer` / 日々の食事判定（Railsのenum管理。`0: too_little`, `1: normal`, `2: too_much` として整数で安全かつ軽量に管理）
- ※ **[複合ユニーク制約]**：`user_id` と `date` の組み合わせに対する制約（1日1レコード制限）により、ボタン連打によるデータ重複を徹底的に防ぎます。

#### 📙 weight_records テーブル（体重推移ノート）
ユーザーが日々の簡易フォーム等から入力・更新したリアルな現在の体重の推移（履歴）を蓄積していくためのテーブルです。
- **id** : `integer` / 主キー（体重記録1件ごとの通し番号）
- **user_id** : `integer` / 外部キー（どのユーザーの記録かを特定。usersテーブルのidと紐付け）
- **date** : `date` / 体重の記録日（折れ線グラフの横軸や、カレンダーマスへの表示用）
- **weight** : `float` / 記録されたその日の体重（kg / 小数点以下の数値にも対応できる浮動小数点型）
- ※ **[複合ユニーク制約]**：`user_id` と `date` の組み合わせに対する制約（1日1レコード制限）を設け、日々の記録が綺麗に1日1件でスッキリ管理できるようにします。

### 💡 データベース設計（正規化）のこだわりと整合性の担保

1. **第3正規形の徹底による冗長性の排除**：
   画面に表示される「標準体重」や「1日の目標カロリー」は、データベース内にはカラムとして保持していません。ユーザーの【性別・年齢・身長・体重】の基本データから、裏側のプログラム（Rails）がアプリ起動時にその場で一瞬で最新の数値を自動計算して画面（React）に伝える設計にしています。これにより、データ変更時のすれ違いバグを防ぎ、無駄な容量を排除したスマートなデータ構造を実現しています。
2. **未来のUX向上（グラフ・カレンダー連携）を見据えたテーブル分離**：
   最新の体重をusersテーブルに単に上書きするのではなく、新しく `weight_records` テーブルを独立させて履歴として蓄積する設計にブラッシュアップしました。これにより、本リリース以降に予定している「体重の折れ線グラフ表示機能」や「カレンダー内への日々の体重数字の併記」といった高度な振り返り機能をスムーズに実装できる拡張性を完全に確保しています。
3. **複合ユニーク制約によるデータ不整合の根絶（食事記録・体重記録ともに導入）**：
   `meal_records` と `weight_records` の両テーブルにおいて、それぞれ `user_id` と `date` の組み合わせに対してデータベース階層で【複合ユニーク制約（1日1レコード制限）】を設けています。これにより、ユーザーの誤操作やボタン連打による日々のデータの重複をシステム側で自動的にシャットアウトし、4日連続の普通判定アラートの正常な稼働や、日々の体重推移の綺麗な管理を100%狂わせない堅牢なガードを構築しています。
4. **適切な1対多（1対N）のリレーションシップ**：
   「1人のユーザー（Users）」に対して、毎日のログが蓄積していく「多くの食事記録（MealRecords）」および「多くの体重記録（WeightRecords）」がそれぞれカラスの足（三股の矢印）の線で適切に結ばれた、無駄のない美しいリレーションを構築しています。

## 💡 本番環境デプロイ時のトラブルシューティング

本番環境（Render / Neon）への初期デプロイ時に発生したエラーと、その解決アプローチの記録です。

### 1. Docker Build Context の重複によるディレクトリ迷子
* **エラー**: `error: invalid local: resolve : lstat ... no such file or directory`
* **原因**: Renderの設定で `Root Directory` と `Docker Build Context` の両方に `backend` を指定したため、Renderが `backend/backend` という存在しない階層を探してしまっていた。
* **解決策**: `Docker Build Context` 側を `.`（カレントディレクトリ）に修正し、コンテキストの重複を解消。

### 2. 本番環境における Vite の無限ポーリング（無限ループ）
* **エラー**: フロントエンドのビルドが終了せず、ログが無限にループする。
* **原因**: 開発環境用のホットリロード設定である `usePolling: true` が本番サーバー上でも動作し続け、プロセスの終了を妨げていた。
* **解決策**: `vite.config.ts` を修正し、本番環境（`process.env.RENDER` が存在する場合）では自動的にポーリングがオフになるよう動的切り替えを実装。

### 3. production環境における secret_key_base の不足
* **エラー**: `ArgumentError: Missing secret_key_base for 'production' environment`
* **原因**: Railsを production モードで安全に起動するために必要な、暗号化用の秘密鍵（合言葉）が本番環境の環境変数に設定されていなかった。
* **解決策**: セキュリティと秘匿性を担保するため、ソースコードには直接記述せず、Renderの `Environment Variables` に `SECRET_KEY_BASE` としてトークンを安全に登録。


## 🤖 GitHub Actions による CI環境（自動テスト・静的解析）の構築

本アプリの品質向上と開発効率化のため、GitHub Actionsを用いた継続的インテグレーション（CI）環境を構築しました。コードを `push` または `pull_request` した際に、仮想環境上で以下の自動チェックが全自動で稼働します。

### 📊 導入した自動チェック機能
* **静的解析（Lint）**: **RuboCop** を用い、コードスタイルの乱れやバグの種、文法エラーを自動検閲。
* **自動テスト**: **RSpec** を用い、機能が期待通りに動作するかを全自動で実行。

### 💡 構築におけるトラブルシューティングの記録

仮想環境（Ubuntu）上の Docker Compose 内でテスト・静的解析を安定して完走させるため、以下の高度なデバッグと環境最適化を行いました。

1. **GitHub Actions runner における Docker Compose コマンドの最適化**
   * **課題**: `docker-compose`（ハイフン繋ぎ）のサポート終了に伴い、仮想環境上で `command not found` (exit code 127) が発生。
   * **解決策**: 2026年現在のGitHub標準仕様に合わせ、ワークフロー定義ファイル（`ci.yml`）内のすべての記述をスペース空けの最新コマンド（`docker compose`）へと完全統一しました。

2. **Gems（依存パッケージ）の同期不全と本番環境除外の回避**
   * **課題**: コンテナ起動直後に `bundler: command not found: rubocop` が発生。Gemsが正しく読み込まれない問題に直面。
   * **原因**: デプロイ用に設定していた `RAILS_ENV=production` の影響により、Railsが気を利かせて開発・テスト用のGems（RuboCopやRSpec）を自動で除外（omit）してビルドしていたことが判明。
   * **解決策**: `ci.yml` 内の環境変数を `development` へと強制上書きし、Gemsのインストール時に `--without production` 設定を動的に解除。これにより、CI環境に必要なすべてのテスト・静的解析ツールを100%確実にコンテナ内へフルインストールさせることに成功しました。

3. **コンテナ及びデータベースの起動待ち（Sleep処理）の同期制御**
   * **課題**: Dockerの起動直後にRuboCopが超高速で実行されてしまい、コンテナの内部プロセスやPostgreSQLの立ち上がりが間に合わず、コマンドを見失うすれ違いバグが発生。
   * **解決策**: `sleep 10` のステップを導入し、コンテナおよびデータベースが完全に「準備完了」になるのを待ってからチェックを走らせる同期制御を実装。これにより100%安定してCIを通過させるロジックを完成させました。

4. **コード品質（Documentationコメント）の完全適合**
   * **課題**: 自動修正（`rubocop -A`）では対応できない、クラス定義時のトップレベル説明コメントの不足（`Style/Documentation`）によりCIがストップ。
   * **解決策**: `config/application.rb` や各主要クラスに適切なインデントで説明コメントを追記。Rubyの構文（Syntax）を崩さないようミリ単位のコードレビューを行い、最終的に **0 offenses（指摘ゼロ）** の完全なクリーン状態を達成し、念願の **Success (緑のチェックマーク 🟢)** を灯らせました。


## 🎨 認証機能（フロントエンド：React）のUI・ルーティング・状態保持の実装

ユーザーが直感的に、かつ安全にアカウント作成、ログイン、ログアウトの操作を行えるよう、日本語表記に統一したスタイリッシュな入力フォーム（UI）および認証状態の永続化システムを実装しました。

### ⚙️ 実装したフロントエンド機能
* **日本語表記の入力フォームUI（Register / Login）**: ユーザーが操作に迷わないようすべての表記を親切な日本語に統一。フォームには外枠（`border`）や影（`boxShadow`）を施し、入力エリアの視認性を劇的に向上させたモダンなデザインを適用。
* **画面切り替えとURL管理（react-router-dom）**: フロントエンド専用のルーティングパッケージを導入し、`/register`（新規登録）や `/login`（ログイン）のURLとReact画面をガチッと紐付け。
* **ブラウザの引き出しによるログイン永続化（localStorage）**: 認証成功時にRailsから届くJWT（デジタル会員証）をブラウザの安全な引き出し（`localStorage`）へ保存。ページ更新時にも自動で状態をチェック（`useEffect`）し、ログイン状態を永続的にキープ。
* **状態に応じた動的UIとクリーンアップ（ログアウト）**: ログイン状態を検知して「ログイン・新規登録」ボタンと「ログアウト」ボタンを自動でパチッと切り替える動的UIを実装。ログアウト時には引き出しから会員証を完全消去（`removeItem`）する安全なクリーンアップ処理を徹底。

### 💡 API連携におけるインフラ・デバッグの記録
